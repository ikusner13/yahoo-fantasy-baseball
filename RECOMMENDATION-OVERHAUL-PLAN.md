# Recommendation Overhaul Plan

Last updated: 2026-04-09
Branch: `feature/recommendation-overhaul`

## Goal

Build a recommendation-first Yahoo fantasy baseball assistant that improves weekly H2H category win odds while preserving championship equity.

The app remains recommendation-only. It should send short, direct Telegram messages that tell the user:

- what to do
- what not to do
- which categories matter
- which categories to ignore
- how confident the app is

The decision engine should be driven primarily by projections, schedule volume, matchup math, and role/news changes. LLMs should support explanation, synthesis, and learning loops, not replace the underlying math.

## Current State

The current app is usable, but the recommendation core is still heuristic-heavy:

- Production orchestration is centralized in `src/gm.ts`.
- Daily recommendations are generated by deterministic lineup, waiver, streaming, and matchup modules.
- The app already has useful context inputs: projections, park factors, platoon data, recent performance, Statcast, Vegas, and Yahoo matchup state.
- A Monte Carlo simulator exists in `src/analysis/monte-carlo.ts`, but production recommendation paths do not use it.
- The “projected” weekly matchup output is not a true projection; it is a current-score heuristic.
- Learning exists as reflections and retrospectives, but not as a hard outcome-based decision scoring system.

Current code worth preserving:

- Yahoo / Telegram / Cloudflare integration
- projection and schedule ingestion
- test harness and simulation entry points
- data matching and caching plumbing
- notification pathways

Current code that should likely be replaced or heavily refactored:

- matchup classification as the primary decision engine
- fixed threshold category weighting
- greedy lineup optimization as the final authority
- waiver logic based mostly on z-score deltas
- streaming logic based mostly on static thresholds
- retrospective logic that does not score specific recommendations against counterfactuals

## Research-Based Recommendation Philosophy

The build should follow this signal hierarchy:

1. Rest-of-season and weekly projections are the baseline.
2. Schedule volume and remaining opportunity are critical in daily H2H.
3. Role changes, lineup spot changes, and closer/news updates are high-priority overrides.
4. Recent performance should be a modest modifier, not the main engine.
5. Statcast / underlying skills should validate breakouts and regression flags.
6. Vegas or run-environment context should be a small tiebreaker only.

For this user’s league:

- exact opponent roster modeling is not a requirement
- live scoreboard state matters more than deep opponent scouting
- the app should surface small edges, but not present weak evidence as strong certainty
- short-term moves are acceptable only if they do not recklessly damage title equity

## Product Requirements

### 1. League-Aware Engine

The app must stop relying on hardcoded assumptions where league settings matter.

It needs a first-class league profile containing:

- roster slots
- scoring categories
- weekly add limit
- waiver style
- innings minimum
- playoff structure

The user’s exact Yahoo league should be the default configured profile.

### 2. True Matchup Win-Odds Engine

Replace current “safe / swing / lost” heuristics as the main decision authority with a probabilistic engine.

The engine should:

- project remaining category outcomes from the live scoreboard
- model remaining opportunities for our active lineup
- evaluate changes in expected category wins and overall matchup win probability
- expose per-category win probabilities
- expose marginal value of a move in win-probability terms

This should use or extend the existing Monte Carlo foundation instead of keeping it isolated.

### 3. Recommendation Types

The app should support these recommendation classes:

- daily lineup optimization
- streamer recommendations
- waiver add/drop recommendations
- watchlist / must-add alerts
- Sunday endgame tactics
- season-long trade ideas only if they improve title equity

Every recommendation should have:

- primary action
- ranked alternatives
- target categories
- categories to ignore
- “do not do this” guidance
- confidence
- why the edge exists

### 4. Confidence Model

Confidence must be quantitative and explainable.

Suggested confidence inputs:

- projected win-probability delta
- category probability swing
- agreement across signals
- data completeness
- uncertainty from rate categories and role ambiguity

Example interpretation:

- High: strong EV edge, clean data, stable role
- Medium: positive edge with moderate uncertainty
- Low: slight edge or noisy evidence

### 5. Watchlist and Must-Add Detection

Add a ranked watchlist system with escalation rules.

It should prioritize:

- newly promoted closers
- SV+H role changes
- major lineup-order moves
- prospects/call-ups with immediate playing time
- free agents whose projected value and weekly fit exceed a threshold

The system should distinguish:

- must add now
- strong watch
- monitor only

### 6. Learning and Evaluation

The app should learn from outcomes, but with hard math first.

Needed additions:

- recommendation ledger with timestamps, rationale, confidence, and predicted delta
- matchup-by-matchup scorecard
- decision hit-rate metrics
- category-level forecast calibration
- retrospective learning summaries generated from the scored outcomes

LLMs should be used to summarize patterns and suggest tuning changes, not to decide whether the raw measurements happened.

## Recommended Architecture

### Recommendation Core

Create a new recommendation layer rather than continuing to grow `gm.ts`-level heuristics.

Suggested modules:

- `src/recommendation/league-profile.ts`
- `src/recommendation/matchup-state.ts`
- `src/recommendation/probabilistic-engine.ts`
- `src/recommendation/decision-evaluator.ts`
- `src/recommendation/confidence.ts`
- `src/recommendation/watchlist.ts`
- `src/recommendation/recommendation-types.ts`
- `src/recommendation/explanations.ts`

### Evaluation Core

Suggested modules:

- `src/evaluation/recommendation-ledger.ts`
- `src/evaluation/backtest.ts`
- `src/evaluation/calibration.ts`
- `src/evaluation/matchup-scorecard.ts`

### Messaging Layer

Keep Telegram delivery, but make it consume normalized recommendations instead of assembling messages directly from heuristics.

The message formatter should accept:

- top recommendation
- ranked options
- confidence
- “why”
- “avoid”

### LLM Role

LLMs should be constrained to:

- compress structured recommendation objects into short direct messages
- summarize retrospective patterns
- explain disagreements between model signals when necessary

LLMs should not:

- invent category math
- override the engine without explicit justification and structured support

## Rewrite Decision

A full application rewrite is not justified yet.

Recommended approach:

- preserve integrations, data ingestion, storage, and delivery paths
- rewrite the recommendation core behind stable interfaces
- phase old heuristic modules out as the new engine takes over

This is effectively a recommendation-engine rewrite, not an app rewrite.

## Implementation Workstreams

### Workstream A: League Profile and Domain Cleanup

Deliverables:

- first-class league profile module
- removal of critical hardcoded league assumptions from recommendation logic
- user-league fixture

Tests:

- unit tests for league profile parsing and defaults
- tests that derived limits and categories match the user’s Yahoo league

Verification:

- no recommendation path should rely on magic numbers for this user’s settings without passing through the league profile

### Workstream B: Probabilistic Matchup Engine

Deliverables:

- production-grade adapter around Monte Carlo / counterfactual evaluation
- per-category win probability outputs
- move delta scoring for lineup, streaming, and waiver decisions

Tests:

- deterministic seeded simulation tests
- counterfactual tests showing better options improve win probability
- edge-case tests for ERA/WHIP/OBP accumulation
- tests for innings-minimum behavior

Verification:

- existing heuristic outputs can be compared against the new engine
- seeded scenarios should be reproducible

### Workstream C: Recommendation Evaluators

Deliverables:

- lineup decision evaluator
- streaming evaluator
- waiver evaluator
- Sunday tactic evaluator

Tests:

- scenario-based golden tests
- regression tests for common H2H endgame situations
- ranked-option ordering tests

Verification:

- every recommendation type should expose a structured delta and confidence

### Workstream D: Watchlist and Alerts

Deliverables:

- ranked watchlist
- must-add escalation rules
- role/news integration layer

Tests:

- role-change scenarios
- closer takeover scenarios
- lineup spot / call-up scenarios
- duplicate suppression tests

Verification:

- must-add alerts should only trigger when thresholds are exceeded

### Workstream E: Confidence, Messaging, and LLM Synthesis

Deliverables:

- confidence scoring module
- structured recommendation formatter
- LLM compression prompts for short direct Telegram output

Tests:

- confidence-banding tests
- formatter tests
- prompt contract tests using fixtures

Verification:

- output must remain short and direct
- output must include action, rationale, and avoid guidance

### Workstream F: Evaluation and Learning

Deliverables:

- recommendation ledger
- decision outcome scoring
- calibration metrics
- backtest / replay support

Tests:

- recommendation logging tests
- outcome scoring tests
- calibration metric tests
- replay consistency tests

Verification:

- app should be measurable day by day and matchup by matchup
- retrospectives should be generated from scored outcomes, not only freeform memory

## Testing Strategy

### Unit Tests

Cover:

- league profile
- matchup probability calculations
- decision delta calculations
- confidence computation
- watchlist thresholds
- evaluation metrics

### Scenario / Golden Tests

Add rich fixtures for:

- protecting ERA/WHIP late in week
- chasing counting stats under innings pressure
- deciding between two similar hitters with different category fit
- deciding whether to burn one of six weekly adds
- must-add closer changes
- Sunday final-day category reallocation

### Integration Tests

Cover:

- Yahoo data -> recommendation objects
- recommendation objects -> Telegram messages
- recommendation ledger -> retrospective output

### Backtesting / Replay

Add a replay mode that:

- rehydrates historical matchup states
- reruns the recommendation engine
- compares predicted edges to actual outcomes where possible

### Verification Gates

Before promoting the new engine to production paths:

- seeded simulation tests must pass
- scenario regression suite must pass
- recommendation objects must include confidence and avoid guidance
- old vs new engine comparison should show equal or better decision quality on curated fixtures

## Rollout Plan

### Phase 1

- introduce league profile
- introduce new probability engine interfaces
- add scenario fixture library
- add evaluation ledger schema / plumbing

### Phase 2

- move Sunday tactics and weekly matchup preview onto the new probability engine
- add confidence and structured recommendation objects
- add watchlist / must-add alerts

### Phase 3

- move lineup, streaming, and waiver recommendations onto counterfactual move evaluation
- keep heuristics only as fallback

### Phase 4

- add backtest / replay and retrospective scoring
- tune thresholds using measured outcomes

## Open Decisions

- Whether trade recommendations should be deferred until the core weekly recommendation engine is stable
- Whether Vegas data remains worth the operational complexity after the new engine is in place
